# Power BI — build guide

Power BI Desktop is a Windows GUI, so the `.pbix` is built by hand (it cannot
be generated by the pipeline). This guide is the spec: follow it and the
dashboard will match the PRD. Ship the finished `.pbix` in this folder plus PNG
screenshots in the root README.

> **Data note:** this dashboard uses simulated SBL data for portfolio
> demonstration only, not real broker, exchange, custodian, or client data.

> **Publishing note:** publishing to Power BI *Service* needs an organisational
> account, so a personal account cannot host a public link. For a portfolio,
> the `.pbix` file + screenshots are the accepted deliverable.

## 1. Connect to data

**Option A — Postgres (primary).** Home → Get Data → PostgreSQL database.
Server `localhost:5432`, Database `sbl`. Import mode. Load the four views:
`v_latest_snapshot`, `v_sector_pressure`, `v_sector_pressure_inputs`,
`v_hard_to_borrow`, and the base table `sbl_daily_metrics` (for time-series
pages).

**Option B — CSV (quick start, no DB).** Get Data → Text/CSV →
`data/processed/sbl_analytics.csv` and `sbl_sector_pressure.csv`.

## 2. Model

- Use a small star schema rather than flat, disconnected view reads. Create
  `Dim Sector` as a dimension table with one row per unique sector value.
- Create many-to-one relationships from each fact/view table into `Dim Sector`:
  `sbl_daily_metrics[sector]`, `v_latest_snapshot[sector]`,
  `v_hard_to_borrow[sector]`, `v_sector_pressure_inputs[sector]`, and
  `v_sector_pressure[sector]` all relate to `Dim Sector[sector]`.
- Set cross-filter direction to single direction: `Dim Sector` filters the
  facts/views. This is a deliberate dimensional-modelling choice, keeping
  slicers and cross-highlighting predictable while avoiding ambiguous
  fact-to-fact filters.
- Mark `date` as a date column; add a simple Date table if you want proper time
  intelligence (`Calendar = CALENDAR(MIN(...), MAX(...))`, relate to `date`).
- Reading from the SQL **views** keeps the DAX below trivial; the logic already
  lives in SQL.

## 3. Core measures (DAX)

```DAX
Total Available Borrow Value = SUM(sbl_daily_metrics[available_borrow_value])
Avg Borrow Fee (bps)         = AVERAGE(sbl_daily_metrics[borrow_fee_bps])
Avg Utilization %            = AVERAGE(sbl_daily_metrics[utilization_pct])
// Hard-to-Borrow  = CALCULATE(DISTINCTCOUNT(sbl_daily_metrics[symbol]), sbl_daily_metrics[hard_to_borrow_flag] = TRUE)
// Declining Avail = CALCULATE(DISTINCTCOUNT(sbl_daily_metrics[symbol]), sbl_daily_metrics[daily_available_change_pct] <= -15)

HTB Colour =
SWITCH(TRUE(),
    SELECTEDVALUE(sbl_daily_metrics[hard_to_borrow_flag]) = TRUE(), "Red",
    [Avg Utilization %] >= 75 || [Avg Borrow Fee (bps)] >= 200, "Amber",
    "Green")
```

> **Threshold note:** the amber display thresholds above (`utilization >= 75`,
> `fee >= 200`) are a second dashboard-only threshold set and do not currently
> live in `src/config.py`. Either document them explicitly in the root README as
> visual display thresholds, or align the dashboard colours with the configured
> model thresholds so they are not left unexplained.

## 4. Pages (match the PRD)

**Page 1 — Market Overview.** KPI cards use `sbl_daily_metrics` and the core
measures above. Sector pressure bar uses `v_sector_pressure[sector]` and
`v_sector_pressure[sector_pressure_score]`. The top table uses
`v_latest_snapshot`: `symbol`, `sector` (displayed as a short label),
`utilization_pct`, `borrow_fee_bps`, `days_to_cover_proxy`,
`hard_to_borrow_flag`, and `short_eligible`. Surface `short_eligible`, or
visually mark short-ineligible names such as ORI/THAI, so viewers do not assume
every listed name is shortable. The trend visual is **Hard-to-Borrow Names
Trend**: count of `sbl_daily_metrics[symbol]` where
`sbl_daily_metrics[hard_to_borrow_flag] = TRUE` by `sbl_daily_metrics[date]`.
Correction to make: because this trend is a count of HTB names, its Y-axis must
be labelled `# Hard-to-Borrow`, not `Avg Utilization %`.

> **Sector pressure is a per-snapshot ranking, not a time series.**
> `sector_pressure_score` min-max scales each component across the 10 sectors on
> the selected date, so the calmest sector is always ≈ 0 and the tightest
> ≈ 100 regardless of absolute conditions. Use it only on the single-date
> bar chart; never on a line/trend visual, and do not compare a sector's score
> across dates. (`htb_ratio` is also quantized: ~3–4 names per sector, so
> one flag is a large step.) For trends over time, chart the raw
> `avg_utilization` / `avg_fee_bps` instead.

**Page 2 — Stock Detail.** Slicer uses `sbl_daily_metrics[symbol]`. Latest
snapshot cards use `v_latest_snapshot`: `utilization_pct`, `borrow_fee_bps`,
`days_to_cover_proxy`, `collateral_requirement_pct`, `hard_to_borrow_flag`, and
`short_eligible`. Line charts use `sbl_daily_metrics[date]` on the X-axis with
`available_borrow_qty`, `borrow_fee_bps`, `utilization_pct`, and
`days_to_cover_proxy` as separate measures/series. The auto risk note text box
uses `v_latest_snapshot[hard_to_borrow_reason]`.

**Page 3 — Hard-to-Borrow Monitor.** Main table uses `v_hard_to_borrow`:
`symbol`, `sector`, `available_borrow_qty`, `borrow_fee_bps`,
`utilization_pct`, `days_to_cover_proxy`, `daily_available_change_pct`,
`daily_fee_change_bps`, `short_eligible`, and `hard_to_borrow_reason`.
Conditional formatting is driven by `HTB Colour` (red flagged / amber watchlist
/ green clear). Short-ineligible flagged names must be visually distinguished
with a separate icon or colour, since borrow tightness is only actionable for
short-sale execution on eligible names. Ranking visuals use `v_latest_snapshot`:
highest `utilization_pct`, largest `daily_fee_change_bps`, largest availability
decline by `daily_available_change_pct`, and highest `days_to_cover_proxy`.

**Page 4 — Sales-Trading Memo.** A card/multi-row card reproduces the generated
`reports/sbl_sales_trading_memo.md` narrative. If rebuilt natively in Power BI,
bind the memo summary to latest-snapshot fields from `v_latest_snapshot`
(`daily_available_change_pct`, `hard_to_borrow_flag`, `daily_fee_change_bps`,
`hard_to_borrow_reason`) and sector pressure from `v_sector_pressure[sector]` /
`v_sector_pressure[sector_pressure_score]`. Keep DAX light; the HTB,
eligibility, and sector-pressure logic already lives in Python/SQL.
